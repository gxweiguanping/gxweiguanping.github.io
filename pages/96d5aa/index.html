# 方式一：直接使用Thread类

  • 直接继承Thread类，重写其run()方法，或者使用lambda表达式作为线程任务。
  • 线程的任务逻辑直接放在run()方法内。
@Slf4j
public class MyThread  {
    public static void main(String[] args) {
        
        Thread t1 = new Thread(() -> {
            // 要执行的任务
            log.debug("要执行的任务");
        });

        t1.start();
    }
}

# 方式二：使用 Runnable 配合 Thread

  • 实现Runnable接口，将任务逻辑放在run()方法中。

@Slf4j
public class MyRunnable {
    public static void main(String[] args) {
        Runnable myRunnable = new Runnable() {
            @Override
            public void run() {
                // 要执行的任务
                log.debug("要执行的任务");
            }
        };
        // 创建并启动线程
        Thread thread = new Thread(myRunnable);
        thread.start();
    }
}

# 方式三：FutureTask 配合 Thread

  • 使用FutureTask包装Callable任务，Callable允许线程返回结果。
  • FutureTask实现了Runnable接口，可被Thread执行。
  • 可以在主线程中通过futureTask.get()获取线程执行结果。
@Slf4j
public class MyCallable {
    public static void main(String[] args) {
        // 创建任务对象
        FutureTask<String> futureTask = new FutureTask<>(() -> {
            return "要执行的任务";
        });

        // 将FutureTask的对象作为参数传递到Thread类的构造器中，创建Thread对象，并调用start()
        new Thread(futureTask).start();

        try {
            String result = futureTask.get();
            log.debug(result);
        } catch (InterruptedException | ExecutionException e) {
            e.printStackTrace();
        }
    }
}

# 小结

  • 方式一对于简单的任务可能更直观，但扩展性较差，因为Java不支持多继承。
  • 方式二更灵活，因为它可以继承其他类，同时实现Runnable接口，支持更好的代码组织和重用，开发中优先选择实现Runnable接口的方式。
  • 方式三实现Callable接口的方式创建线程的强大之处
    • call()可以有返回值的
    • call()可以抛出异常，被外面的操作捕获，获取异常的信息
    • Callable是支持泛型的

# Runnable和Callable抛出异常验证

Callable接口的call()可以抛出异常，被外面的操作捕获，获取异常的信息，原因是Callable的call() throws Exception，而Java里面有这样一条规定，子类不能抛出比父类更多的异常（包括范围更大的异常）

public class CallableDemo {
    public static void main(String[] args) {
        // 创建Callable任务
        Callable<String> callableTask = () -> {
            // 抛出异常
            throw new IllegalStateException("发生异常啦！");
        };

        // 使用FutureTask包装Callable任务
        FutureTask<String> futureTask = new FutureTask<>(callableTask);

        // 创建并启动线程
        Thread thread = new Thread(futureTask);
        thread.start();

        try {
            // 获取线程执行结果
            String result = futureTask.get();
            System.out.println("线程执行结果: " + result);
        } catch (InterruptedException | ExecutionException e) {
            // 捕获线程中抛出的异常
            System.err.println("捕获到异常: " + e.getMessage());
        }
    }
}

# 小结

  • start() 方法：当调用 start() 方法时，会启动一个新线程，并将线程状态从新建状态（NEW）转变为就绪状态（RUNNABLE）。此后，线程调度器可以选择合适的时机让该线程执行，并在执行时调用其 run() 方法。
  • run() 方法：直接调用 run() 方法并不会启动新线程，只是在当前线程中执行 run() 方法的代码块。因此，线程状态并不会改变，仍然是当前线程的状态，比如如果在主线程中调用了 run() 方法，线程状态仍然是主线程的状态（通常是运行状态）。

# sleep 与 yield

# sleep()

  1. 调用 Thread.sleep(long millis) 会让当前线程从 RUNNING 进入 TIMED_WAITING 状态（阻塞）。这意味着该线程在指定的毫秒数内不会执行任何操作，并且不会消耗CPU时间。

  2. 其他线程可以使用 interrupt() 方法打断正在睡眠的线程。如果一个线程在睡眠期间被中断，它会抛出一个 InterruptedException 异常。你可以捕获这个异常并决定如何处理，例如恢复任务或者优雅地终止程序。

  3. 当睡眠结束时，线程并不会立即得到执行。线程调度器会选择下一个要运行的线程，这可能不是刚刚醒来（从 TIMED_WAITING 变为 RUNNABLE）的线程。因此，即使睡眠时间结束了，线程也可能会等待一段时间才能再次开始执行。

  4. 建议用 TimeUnit 的 sleep 代替 Thread 的 sleep 来获得更好的可读性

    TimeUnit.MINUTES.sleep(4);  // sleeping for 4 minutes

# yield()

  1. 调用 Thread.yield() 会让当前线程从 RUNNING 进入 RUNNABLE 就绪状态**。提示当前线程让出执行机会**，这意味着当前线程不再主动要求执行，而是让其他同样处于就绪状态的线程有机会执行。
  2. 实际上，yield() 方法的具体实现依赖于操作系统的任务调度器。不同操作系统的调度器可能会有不同的行为。有些调度器可能会忽略 yield() 的提示，而有些调度器可能会优先选择其他就绪线程。
public class YieldExample {
    public static void main(String[] args) {
        Thread highPriorityThread = new Thread(new Runnable() {
            @Override
            public void run() {
                while (true) {
                    System.out.println("High priority thread running.");
                    Thread.yield(); // 提示当前线程让出执行机会
                }
            }
        }, "High Priority");
        
        highPriorityThread.setPriority(Thread.MAX_PRIORITY);
        
        Thread lowPriorityThread = new Thread(new Runnable() {
            @Override
            public void run() {
                while (true) {
                    System.out.println("Low priority thread running.");
                }
            }
        }, "Low Priority");
        
        lowPriorityThread.setPriority(Thread.MIN_PRIORITY);
        
        highPriorityThread.start();
        lowPriorityThread.start();
    }
}

# join等待方法详解

在Java多线程编程中，join()方法是Thread类的一个成员方法，它允许一个线程等待另一个线程完成其执行。简单来说，当在一个线程上调用另一个线程的join()方法时，调用线程会暂停执行，直到被调用线程执行完毕。

# join()方法的签名和使用

public final void join()
public final void join(long millis)
public final void join(long millis, int nanos)
1
2
3

这些是join()方法的不同重载版本，用于指定不同的等待时间：

  1. join()：阻塞当前线程，直到被调用join()的线程结束。
  2. join(long millis)：阻塞当前线程最多不超过给定的毫秒数，如果被调用join()的线程在此之前已经结束，则立即返回。
  3. join(long millis, int nanos)：阻塞当前线程最多不超过给定的毫秒数和纳秒数，如果被调用join()的线程在此之前已经结束，则立即返回。

@Slf4j
public class TestJoin {
    static int result = 0;

    public static void main(String[] args) throws InterruptedException {
        testJoin();
    }

    public static void testJoin() throws InterruptedException {
        Thread t1 = new Thread(() -> {
            try {
                // 休眠1毫秒
                Thread.sleep(1);
                result = 10;
            } catch (InterruptedException e) {
                e.printStackTrace();
            }

# interrupt中断方法详解

# 什么是中断机制

一个线程不应该由其他线程来强制中断或停止，而是应该由线程自己自行停止，自己来决定自己的命运。 所以，Thread.stop()、Thread.suspend()、Thread.resume()等关于强制线程停止的方法都已经废弃了。

在Java中，没有办法立即停止一个线程，然而停止线程本身这个功能是很重要的，比如说取消一个耗时的操作。

因此，Java提供了一种用于停止线程的协商机制：中断，也即中断标识协商机制。

Java的中断只是一种协商机制，Java并没有给中断增加任何语法，中断的过程完全需要程序员自己来实现。 若要中断一个线程，我们需要手动调用该线程的interrupt方法，该方法也仅仅是将线程对象的中断标识设为true。接着我们需要自己写代码不断地在程序每一个关键点检测当前线程的标识位，如果为true，表示别的线程请求中断这条线程，此时究竟该做些什么，需要我们自己写代码实现。

每个线程对象中都有一个中断标识位，用于标识线程是否被中断；该标识位为true表示中断，为false表示未中断；通过调用线程对象的interrupt()方法将该线程的中断标识位设为true，可以在别的线程中调用，也可以在自己的线程中调用。

# 场景

如果线程需要执行一个长时间任务，就可能需要能中断线程。中断线程就是其他线程给该线程发一个信号，该线程收到信号后结束执行run()方法，使得自身线程能立刻结束运行。

我们举个栗子：假设从网络下载一个100M的文件，如果网速很慢，用户等得不耐烦，就可能在下载过程中点“取消”，这时，程序就需要中断下载线程的执行。

# 中断API

interrupt() 是 Thread 类的一个实例方法，用于中断当前线程。当一个线程通过 interrupt() 方法被中断时，它会收到一个中断信号，但不会立即停止执行。具体的中断响应需要在线程的代码中进行处理。

isInterrupted() 是 Thread 类的另一个实例方法，用于检查线程的中断状态。通过调用该方法，可以判断线程是否被中断，并根据需要执行相应的逻辑。

static boolean interrupted()：检查当前线程是否被中断，该方法在调用后还会清除该线程的中断状态。

# 打断sleep，wait，join 的线程

这几个方法都会让线程进入阻塞状态 ，打断时会会清空打断状态

以打断 sleep 的线程为例：当调用Thread.sleep(long millis)时，当前线程将暂停执行指定的时间（以毫秒为单位）。如果在这个过程中其他线程调用了这个线程的interrupt()方法，那么sleep()方法会立即抛出一个InterruptedException异常，并且清空线程的中断标志位。你可以捕获这个异常，并根据需要处理中断请求

public class InterruptSleepExample {
    private static final Logger log = LoggerFactory.getLogger(InterruptSleepExample.class);

    public static void main(String[] args) {
        test1();
    }

    private static void test1() {
        Thread t1 = new Thread(() -> {
            try {
                Thread.sleep(5);
            } catch (InterruptedException e) {
                throw new RuntimeException(e);
            }
        }, "t1");
        t1.start();
        try {
            Thread.sleep(1);
        } catch (InterruptedException e) {
            throw new RuntimeException(e);
        }
        t1.interrupt();
        log.debug(" 打断状态: {}", t1.isInterrupted());
    }
}

# 错误思路

  • 使用线程对象的 stop() 方法停止线程

    • stop 方法会真正杀死线程，如果这时线程锁住了共享资源，那么当它被杀死后就再也没有机会释放锁，其它线程将永远无法获取锁

  • 使用 System.exit(int) 方法停止线程

    • 目的仅是停止一个线程，但这种做法会让整个程序都停止

    public void stop() {

# 主线程与守护线程

Java程序入口就是由JVM启动main线程，main线程又可以启动其他线程。当所有线程都运行结束时，JVM退出，进程结束。

如果有一个线程没有退出，JVM进程就不会退出。所以，必须保证所有线程都能及时结束。

但是有一种线程的目的就是无限循环，例如，一个定时触发任务的线程：

class TimerThread extends Thread {
    @Override
    public void run() {
        while (true) {
            System.out.println(LocalTime.now());
            try {
                Thread.sleep(1000);
            } catch (InterruptedException e) {
                break;
            }
        }
    }
}

如果这个线程不结束，JVM进程就无法结束。问题是，由谁负责结束这个线程？

然而这类线程经常没有负责人来负责结束它们。但是，当其他线程结束时，JVM进程又必须要结束，怎么办？

答案是使用守护线程（Daemon Thread）。

守护线程是指为其他线程服务的线程。在JVM中，所有非守护线程都执行完毕后，无论有没有守护线程，虚拟机都会自动退出。

因此，JVM退出时，不必关心守护线程是否已结束。

如何创建守护线程呢？方法和普通线程一样，只是在调用start()方法前，调用setDaemon(true)把该线程标记为守护线程：

Thread t = new MyThread();
// 设置为守护线程
t.setDaemon(true);
t.start();

在守护线程中，需要注意：

  • 守护线程不能持有任何需要关闭的资源，例如打开文件等，因为虚拟机退出时，守护线程没有任何机会来关闭文件，这会导致数据丢失。
  • 垃圾回收器线程就是一种守护线程 。
  • Tomcat 中的 Acceptor 和 Poller 线程都是守护线程，所以 Tomcat 接收到 shutdown 命令后，不会等待它们处理完当前请求 。

# 五种状态 ( 操作系统 层面)

  • 【初始状态】仅是在语言层面创建了线程对象，还未与操作系统线程关联

  • 【可运行状态】（就绪状态）指该线程已经被创建（与操作系统线程关联），可以由 CPU 调度执行

  • 【运行状态】指获取了 CPU 时间片运行中的状态

    • 当 CPU 时间片用完，会从【运行状态】转换至【可运行状态】，会导致线程的上下文切换

  • 【阻塞状态】

    • 如果调用了阻塞 API，如 BIO 读写文件，这时该线程实际不会用到 CPU，会导致线程上下文切换，进入【阻塞状态】
    • 等 BIO 操作完毕，会由操作系统唤醒阻塞的线程，转换至【可运行状态】
    • 与【可运行状态】的区别是，对【阻塞状态】的线程来说只要它们一直不唤醒，调度器就一直不会考虑调度它们

  • 【终止状态】表示线程已经执行完毕，生命周期已经结束，不会再转换为其它状态

# 六种状态(Thread.State)

从 Java API 层面来描述的 ，根据 Thread.State 枚举，分为六种状态

状态 说明
NEW 初始状态：线程被创建，但还没有调用start()方法
RUNNABLE 运行状态：Java线程将操作系统中的就绪和运行两种状态笼统的称作“运行”
BLOCKED 阻塞状态：表示线程阻塞于锁
WAITING 等待状态：表示线程进入等待状态，进入该状态表示当前线程需要等待
其他线程做出一些特定动作（通知或中断）
TIME_WAITING 超时等待状态：该状态不同于WAITING，它是可以在指定的时间自行返回的
TERMINATED 终止状态：表示当前线程已经执行完毕
