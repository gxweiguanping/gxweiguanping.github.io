# 死信队列

# 死信的概念

RabbitMQ 死信队列（Dead Letter Queue，DLQ）是一种特殊的消息队列，用于处理无法被消费者成功处理的消息。当消息在正常队列中变成死信时，它会被发送到死信队列，而不是被丢弃或无限循环处理。

# 应用场景

为了保证订单业务的消息数据不丢失，需要使用到 RabbitMQ 的死信队列机制，当消息消费发生异常时，将消息投入死信队列中。

例如：对于未支付的订单，在一定时间内未完成支付，我们可以将这些订单消息投递到死信队列。通过设定消息的过期时间，系统可以自动检测并处理这些超时未支付的订单，避免长时间占用资源。

# 死信的来源

触发条件 解释
消息 TTL 过期 消息在队列中存活时间超过 TTL 设定的生存时间后，会成为死信。
队列达到最大长度 当队列中的消息数量达到最大值，无法再继续添加消息时，剩余消息成为死信。
消息被拒绝 当消息被消费者拒绝接收并且不再重新入队（requeue = false），该消息成为死信。

package cn.deadqueue;

import cn.utils.RabbitMQUtils;
import com.rabbitmq.client.BuiltinExchangeType;
import com.rabbitmq.client.Channel;
import com.rabbitmq.client.Connection;
import com.rabbitmq.client.DeliverCallback;

import java.util.HashMap;
import java.util.Map;

/**
 * @Description
 * @Author MrWei
 * @Date 2023/8/17 21:13
 **/
public class Consumer01 {
    //普通交换机名称
    private static final String NORMAL_EXCHANGE = "normal_exchange";
    //死信交换机名称
    private static final String DEAD_EXCHANGE = "dead_exchange";

    public static void main(String[] args) throws Exception {
        Connection connection = RabbitMQUtils.getConnection();
        Channel channel = connection.createChannel();

        //声明死信和普通交换机 类型为 direct
        channel.exchangeDeclare(NORMAL_EXCHANGE, BuiltinExchangeType.DIRECT);
        channel.exchangeDeclare(DEAD_EXCHANGE, BuiltinExchangeType.DIRECT);

        //声明死信队列
        String deadQueue = "dead-queue";
        channel.queueDeclare(deadQueue, false, false, false, null);
        //死信队列绑定：队列、交换机、路由键（routingKey）
        channel.queueBind(deadQueue, DEAD_EXCHANGE, "xiaoming");

        //正常队列绑定死信队列信息
        Map<String, Object> params = new HashMap<>();
        //正常队列设置死信交换机 参数 key 是固定值
        params.put("x-dead-letter-exchange", DEAD_EXCHANGE);
        //正常队列设置死信 routing-key 参数 key 是固定值
        params.put("x-dead-letter-routing-key", "xiaoming");
        params.put("x-max-length",6);

        //正常队列
        String normalQueue = "normal-queue";
        channel.queueDeclare(normalQueue, false, false, false, params);
        channel.queueBind(normalQueue, NORMAL_EXCHANGE, "xiaowang");

        System.out.println("等待接收消息........... ");
        DeliverCallback deliverCallback = (consumerTag, delivery) -> {
            String message = new String(delivery.getBody(), "UTF-8");
            System.out.println("Consumer01 接收到消息" + message);
        };
        channel.basicConsume(normalQueue, true, deliverCallback, consumerTag -> {
        });
    }
}

假设有一个名为"normal-queue"的队列，用于处理订单消息。当订单消息被拒绝时，可以将该被拒绝的消息发送到一个名为"dead-queue"的死信队列。

package cn.deadqueue;

import cn.utils.RabbitMQUtils;
import com.rabbitmq.client.BuiltinExchangeType;
import com.rabbitmq.client.Channel;
import com.rabbitmq.client.Connection;
import com.rabbitmq.client.DeliverCallback;

import java.util.HashMap;
import java.util.Map;

/**
 * @Description
 * @Author MrWei
 * @Date 2023/8/17 21:13
 **/
public class Consumer01 {
    //普通交换机名称
    private static final String NORMAL_EXCHANGE = "normal_exchange";
    //死信交换机名称
    private static final String DEAD_EXCHANGE = "dead_exchange";

    public static void main(String[] args) throws Exception {
        Connection connection = RabbitMQUtils.getConnection();
        Channel channel = connection.createChannel();

        //声明死信和普通交换机 类型为 direct
        channel.exchangeDeclare(NORMAL_EXCHANGE, BuiltinExchangeType.DIRECT);
        channel.exchangeDeclare(DEAD_EXCHANGE, BuiltinExchangeType.DIRECT);

        //声明死信队列
        String deadQueue = "dead-queue";
        channel.queueDeclare(deadQueue, false, false, false, null);
        //死信队列绑定：队列、交换机、路由键（routingKey）
        channel.queueBind(deadQueue, DEAD_EXCHANGE, "lisi");

        //正常队列绑定死信队列信息
        Map<String, Object> params = new HashMap<>();
        //正常队列设置死信交换机 参数 key 是固定值
        params.put("x-dead-letter-exchange", DEAD_EXCHANGE);
        //正常队列设置死信 routing-key 参数 key 是固定值
        params.put("x-dead-letter-routing-key", "lisi");

        //正常队列
        String normalQueue = "normal-queue";
        channel.queueDeclare(normalQueue, false, false, false, params);
        channel.queueBind(normalQueue, NORMAL_EXCHANGE, "xiaowang");

        System.out.println("等待接收消息........... ");

        DeliverCallback deliverCallback = (consumerTag, delivery) -> {
            String message = new String(delivery.getBody(), "UTF-8");
            if (message.equals("info5")) {
                System.out.println("Consumer01 接收到消息" + message + "并拒绝签收该消息");
                //requeue 设置为 false 代表拒绝重新入队 该队列如果配置了死信交换机将发送到死信队列中
                channel.basicReject(delivery.getEnvelope().getDeliveryTag(), false);
            } else {
                System.out.println("Consumer01 接收到消息" + message);
                channel.basicAck(delivery.getEnvelope().getDeliveryTag(), false);
            }

        };
        //开启手动应答
        channel.basicConsume(normalQueue, false, deliverCallback, consumerTag -> {
        });
    }
}

# 延迟队列

# 延迟的概念

延迟队列是一种具有特定属性的队列，其中的元素按照一定的顺序排列。其主要特点在于，元素在一定的时间间隔内并不会立即被取出和处理，而是会在预定的时间到来之后被取出和处理。可以将延迟队列看作一个待处理的时间轴，其中的元素会在指定时间之后被逐个提取出来进行相应的操作。

在实际应用中，延迟队列通常用来处理需要在未来某个特定时间点执行的任务或操作。这些任务可能涉及定时任务、订单超时处理、消息重试等场景。通过将任务放入延迟队列，可以确保它们在预定的时间点得到处理，从而实现精确控制和管理。

# 应用场景

  1. 订单在十分钟之内未支付则自动取消
  2. 新创建的店铺，如果在十天内都没有上传过商品，则自动发送消息提醒
  3. 用户注册成功后，如果三天内没有登陆则进行短信提醒
  4. 用户发起退款，如果三天内没有得到处理则通知相关运营人员
  5. 预定会议后，需要在预定的时间点前十分钟通知各个与会人员参加会议

# TTL的两种设置

# TTL概念

TTL（Time To Live，生存时间）是 RabbitMQ 中用于控制消息或队列在存活期间的一种属性。通过设置 TTL，您可以指定消息在一定时间内保持有效，超过指定时间后消息会被自动移除或丢弃。TTL 可以应用于单个消息，也可以应用于整个队列。

# TTL 在 RabbitMQ 中的应用

消息级别的 TTL： 对于单个消息，您可以为其设置 TTL 属性，以指定消息在一定时间内的有效期。如果消息在 TTL 时间内没有被消费者消费，那么消息将被标记为过期并从队列中移除。这对于具有时间敏感性的任务非常有用，例如验证码短信、临时数据等。

队列级别的 TTL： 您可以在声明队列时为队列设置 TTL 属性，这将影响队列中所有消息的存活时间。在这种情况下，如果队列中的消息在 TTL 时间内没有被消费者消费，那么整个队列会被标记为过期并被自动删除。这对于临时队列、缓存队列等场景可以有效地控制队列的存活时间。

# TTL 的设置方式

消息属性设置： 在发布消息时，可以为消息设置一个 TTL 属性，以毫秒为单位指定消息的有效期。例如，可以在消息的 AMQP Basic Properties 中设置 expiration 字段。

import com.rabbitmq.client.Connection;
import com.rabbitmq.client.ConnectionFactory;
import com.rabbitmq.client.Channel;

public class TTLExample {
    private final static String QUEUE_NAME = "my-queue";
    private final static String HOST = "localhost";
    private static final int PORT = 5672;
    private static final String USERNAME = "guest";
    private static final String PASSWORD = "guest";

    public static void main(String[] argv) throws Exception {
        ConnectionFactory factory = new ConnectionFactory();
        factory.setHost(HOST);
        factory.setPort(PORT);
        factory.setUsername(USERNAME);
        factory.setPassword(PASSWORD);

        try (Connection connection = factory.newConnection();
            Channel channel = connection.createChannel()) {
            channel.queueDeclare(QUEUE_NAME, false, false, false, null);

            // 设置消息的 TTL 为 5000 毫秒（5秒）
            AMQP.BasicProperties properties = new AMQP.BasicProperties.Builder()
                    .expiration("5000") // 设置 TTL
                    .build();

            String message = "Hello, TTL Example!";
            channel.basicPublish("", QUEUE_NAME, properties, message.getBytes());
            System.out.println(" [x] Sent '" + message + "'");
        }
    }
}

队列属性设置： 在声明队列时，可以使用 x-message-ttl 参数设置队列的默认消息 TTL。此参数将影响所有进入该队列的消息。

import com.rabbitmq.client.Channel;
import com.rabbitmq.client.Connection;
import com.rabbitmq.client.ConnectionFactory;
import java.io.IOException;
import java.util.HashMap;
import java.util.Map;
import java.util.concurrent.TimeoutException;

public class SetQueueTTLExample {
    private static final String QUEUE_NAME = "my_queue";
    private static final String HOST = "localhost";
    private static final int PORT = 5672;
    private static final String USERNAME = "guest";
    private static final String PASSWORD = "guest";

    public static void main(String[] args) throws IOException, TimeoutException {
        ConnectionFactory factory = new ConnectionFactory();
        factory.setHost(HOST);
        factory.setPort(PORT);
        factory.setUsername(USERNAME);
        factory.setPassword(PASSWORD);

        try (Connection connection = factory.newConnection();
             Channel channel = connection.createChannel()) {

            // Declare queue with TTL
            Map<String, Object> argsMap = new HashMap<>();
            argsMap.put("x-message-ttl", 60000); // TTL in milliseconds
            channel.queueDeclare(QUEUE_NAME, true, false, false, argsMap);

            System.out.println("Queue with TTL declared successfully.");
        }
    }
}

死信队列中的 TTL： 在使用死信队列（Dead Letter Queue）时，消息在被重新投递到原始队列之前可以设置一个 TTL，以控制重新投递的时间间隔。

import com.rabbitmq.client.Channel;
import com.rabbitmq.client.Connection;
import com.rabbitmq.client.ConnectionFactory;
import java.io.IOException;
import java.util.HashMap;
import java.util.Map;
import java.util.concurrent.TimeoutException;

public class SetMessageTTLExample {
    private static final String QUEUE_NAME = "my_queue";
    private static final String EXCHANGE_NAME = "my_exchange";
    private static final String DEAD_LETTER_EXCHANGE_NAME = "my_dead_letter_exchange";
    private static final String DEAD_LETTER_QUEUE_NAME = "my_dead_letter_queue";
    private static final String HOST = "localhost";
    private static final int PORT = 5672;
    private static final String USERNAME = "guest";
    private static final String PASSWORD = "guest";

    public static void main(String[] args) throws IOException, TimeoutException {
        ConnectionFactory factory = new ConnectionFactory();
        factory.setHost(HOST);
        factory.setPort(PORT);
        factory.setUsername(USERNAME);
        factory.setPassword(PASSWORD);

        try (Connection connection = factory.newConnection();
             Channel channel = connection.createChannel()) {

            // 声明死信交换器和队列
            channel.exchangeDeclare(DEAD_LETTER_EXCHANGE_NAME, "direct");
            channel.queueDeclare(DEAD_LETTER_QUEUE_NAME, true, false, false, null);
            channel.queueBind(DEAD_LETTER_QUEUE_NAME, DEAD_LETTER_EXCHANGE_NAME, "");

            // 声明常规交换器和队列
            channel.exchangeDeclare(EXCHANGE_NAME, "direct");

            // 声明带有 TTL 和死信交换器的队列
            Map<String, Object> argsMap = new HashMap<>();
            argsMap.put("x-message-ttl", 60000); // TTL 毫秒为单位
            argsMap.put("x-dead-letter-exchange", DEAD_LETTER_EXCHANGE_NAME);
            channel.queueDeclare(QUEUE_NAME, true, false, false, argsMap);
            channel.queueBind(QUEUE_NAME, EXCHANGE_NAME, "");

            // 发布带有 TTL 的消息
            String message = "你好，TTL！";
            channel.basicPublish(EXCHANGE_NAME, "", null, message.getBytes());

            System.out.println("带有 TTL 的消息已发布。");
        }
    }
}

# TTL 的注意事项和应用场景

在设置了队列的TTL（Time-To-Live）属性后，一旦消息的存活时间超过了设定的TTL时间，消息会被丢弃或者发送到死信队列中，具体取决于是否配置了死信队列。

然而，第二种方式中，消息的过期并不意味着立即被丢弃。消息是否过期是在即将投递给消费者之前进行判定的。如果当前队列有严重的消息积压情况，已经过期的消息可能会存活较长时间，直到有消费者可以处理它们。

这种行为是由消息队列的实现方式决定的，不同的消息队列可能会有不同的策略来处理过期消息。因此，在使用消息队列时，需要根据具体的需求和实现方式来确定是否需要设置TTL属性以及如何处理过期消息。

  • TTL 不仅适用于队列中的消息，还可以在某些情况下应用于队列本身，以便控制队列的自动删除时间。
  • 使用 TTL 可以帮助您管理过期数据，清理不再需要的消息或队列，从而优化资源利用。
  • 在配置 TTL 时，需谨慎设置过短的时间，以免导致过早地移除消息或队列。
  • TTL 适用于许多场景，如临时数据处理、缓存、定时任务等，能够增加消息的可控性和应用的灵活性。
  • 如果不设置 TTL，表示消息永远不会过期。
  • 如果将 TTL 设置为 0，则表示除非此时可以直接投递该消息到消费者，否则该消息将会被丢弃。

import org.springframework.amqp.core.*;
import org.springframework.beans.factory.annotation.Qualifier;
import org.springframework.context.annotation.Bean;

 * @Description
 * @Author MrWei
 * @Date 2023/8/18 20:35

@Configuration
public class TtlQueueConfig {
    public static final String X_EXCHANGE = "X";
    public static final String QUEUE_A = "QA";
    public static final String QUEUE_B = "QB";
    //死信交换机
    public static final String Y_DEAD_LETTER_EXCHANGE = "Y";
    //死信队列
    public static final String DEAD_LETTER_QUEUE = "QD";

    // 声明 xExchange
    @Bean("xExchange")
    public DirectExchange xExchange() {
        return new DirectExchange(X_EXCHANGE);
    }

    // 声明 死信队列交换机
    @Bean("yExchange")
    public DirectExchange yExchange() {

    }

    public Queue queueA() {

        args.put("x-dead-letter-exchange", Y_DEAD_LETTER_EXCHANGE);
        //声明当前队列的死信路由 key
        args.put("x-dead-letter-routing-key", "YD");
        //声明队列的 TTL
        args.put("x-message-ttl", 10000);
        return QueueBuilder.durable(QUEUE_A).withArguments(args).build();
    }

    // 声明队列 A 绑定 X 交换机
    @Bean
    public Binding queueaBindingX(@Qualifier("queueA") Queue queueA,
                                  @Qualifier("xExchange") DirectExchange xExchange) {
        return BindingBuilder.bind(queueA).to(xExchange).with("XA");
    }

    //声明队列 B ttl 为 40s 并绑定到对应的死信交换机
    @Bean("queueB")
    public Queue queueB() {
        Map<String, Object> args = new HashMap<>(3);

        args.put("x-dead-letter-exchange", Y_DEAD_LETTER_EXCHANGE);
        //声明当前队列的死信路由 key
        args.put("x-dead-letter-routing-key", "YD");
        //声明队列的 TTL
        args.put("x-message-ttl", 40000);
        return QueueBuilder.durable(QUEUE_B).withArguments(args).build();
    }

    //声明队列 B 绑定 X 交换机
    @Bean
    public Binding queuebBindingX(@Qualifier("queueB") Queue queue1B,
                                  @Qualifier("xExchange") DirectExchange xExchange) {
        return BindingBuilder.bind(queue1B).to(xExchange).with("XB");
    }

    //声明死信队列 QD
    @Bean("queueD")
    public Queue queueD() {
        return new Queue(DEAD_LETTER_QUEUE);
    }

    //声明死信队列 QD 绑定关系
    @Bean
    public Binding deadLetterBindingQAD(@Qualifier("queueD") Queue queueD,
                                        @Qualifier("yExchange") DirectExchange yExchange) {
        return BindingBuilder.bind(queueD).to(yExchange).with("YD");
    }

如果使用在消息属性上设置 TTL 的方式，消息可能并不会按时“死亡“

尽管在消息属性上设置了TTL，但是在RabbitMQ中，只会检查第一个消息是否过期并将其丢到死信队列。这意味着如果第一个消息的延时时长很长，而第二个消息的延时时长很短，第二个消息并不会优先得到执行。因此，即使第二个消息设置了延时2秒，它也会在第一个消息执行完之后才会被执行。这就是为什么第二个消息会延迟执行的原因。

配置文件类

在我们自定义的交换机中，这是一种新的交换类型，该类型消息支持延迟投递机制消息传递后并不会立即投递到目标队列中，而是存储在 mnesia(一个分布式数据系统)表中，当达到投递时间时，才投递到目标队列中。

package cn.ttl.config;

import org.springframework.amqp.core.Binding;
import org.springframework.amqp.core.BindingBuilder;
import org.springframework.amqp.core.CustomExchange;
import org.springframework.amqp.core.Queue;
import org.springframework.beans.factory.annotation.Qualifier;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

import java.util.HashMap;
import java.util.Map;

/**
 * @Description
 * @Author MrWei
 * @Date 2023/8/18 22:01
 **/
@Configuration
public class DelayedQueueConfig {
    public static final String DELAYED_QUEUE_NAME = "delayed.queue";
    public static final String DELAYED_EXCHANGE_NAME = "delayed.exchange";
    public static final String DELAYED_ROUTING_KEY = "delayed.routingkey";

    @Bean
    public Queue delayedQueue() {
        return new Queue(DELAYED_QUEUE_NAME);
    }

    //自定义交换机 我们在这里定义的是一个延迟交换机
    @Bean
    public CustomExchange delayedExchange() {
        Map<String, Object> args = new HashMap<>();
        //设置了一个参数x-delayed-type，它的值是direct，表示延迟消息交换机的类型是直连型。
        args.put("x-delayed-type", "direct");
        return new CustomExchange(DELAYED_EXCHANGE_NAME, "x-delayed-message", true, false, args);
    }

    @Bean
    public Binding bindingDelayedQueue(@Qualifier("delayedQueue") Queue queue,
                                       @Qualifier("delayedExchange") CustomExchange delayedExchange) {
        return BindingBuilder.bind(queue).to(delayedExchange).with(DELAYED_ROUTING_KEY).noargs();
    }

}

# 总结

延时队列是在需要延时处理的场景下非常有用的工具。使用RabbitMQ来实现延时队列可以充分利用其特性，如可靠的消息发送和投递，以及死信队列来确保消息至少被消费一次，并且未被正确处理的消息不会丢失。此外，通过RabbitMQ集群的特性，可以解决单点故障问题，确保延时队列的可用性和消息的安全性。

当然，除了RabbitMQ，还有其他选择来实现延时队列，比如使用Java的DelayQueue、Redis的zset、Quartz或者Kafka的时间轮等。每种方式都有其特点和适用的场景，根据具体需求选择最合适的实现方式。
