# 直连模型

在上图的模型中，有以下概念：

P：生产者，也就是要发送消息的程序

C：消费者：消息的接受者，会一直等待消息到来。

queue：消息队列，图中红色部分。类似一个邮箱， 可以缓存消息；生产者向其中投递消息，消费者从其中取出消息。

前置条件：需要先创建/test01虚拟主机，然后将/test01和用户test01绑定。

package helloword;

import com.rabbitmq.client.Channel;
import com.rabbitmq.client.Connection;
import com.rabbitmq.client.MessageProperties;
import utils.RabbitMQUtils;

import java.io.IOException;
import java.util.concurrent.TimeoutException;

/**
 * @Description
 * @Author MrWei
 * @Date 2023/8/15 21:45
 **/
public class Provider {
    /**
     * 生产消息
     *
     * @param args
     * @throws IOException
     * @throws TimeoutException
     */
    public static void main(String[] args) throws IOException, TimeoutException {
        // 获取连接对象
        Connection connection = RabbitMQUtils.getConnection();

        // 创建通道
        Channel channel = connection.createChannel();

        // 通道绑定对应消息队列
        // 参数1:  队列名称 如果队列不存在自动创建
        // 参数2:  用来定义队列特性是否要持久化 true 持久化队列   false 不持久化

        // 发布消息
        // 参数1: 交换机名称
        // 参数2:队列名称
        // 参数3:传递消息额外设置
        // 参数4:消息的具体内容
        channel.basicPublish("", "hello", MessageProperties.PERSISTENT_TEXT_PLAIN, "hello rabbitmq".getBytes());
    }
}

package helloword;

import com.rabbitmq.client.*;
import utils.RabbitMQUtils;

import java.io.IOException;
import java.util.concurrent.TimeoutException;

/**
 * @Description
 * @Author MrWei
 * @Date 2023/8/15 22:09
 **/
public class Consumer {
    /**
     * 消费消息
     *
     * @param args
     * @throws IOException
     * @throws TimeoutException
     */
    public static void main(String[] args) throws IOException, TimeoutException {
        // 获取连接对象
        Connection connection = RabbitMQUtils.getConnection();
        // 创建通道
        Channel channel = connection.createChannel();

        // 通道绑定对应消息队列
        // 参数1:  队列名称 如果队列不存在自动创建
        // 参数2:  用来定义队列特性是否要持久化 true 持久化队列   false 不持久化
        // 参数3:  exclusive 是否独占队列  true 独占队列   false  不独占
        // 参数4:  autoDelete: 是否在消费完成后自动删除队列  true 自动删除  false 不自动删除
        // 参数5:  额外附加参数
        channel.queueDeclare("hello", true, false, false, null);

        // 消费消息
        // 参数1:消费那个队列的消息队列名称
        // 参数2:开始消息的自动确认机制
        // 参数3:消费时的回调接口
        channel.basicConsume("hello",
                true,
                new DefaultConsumer(channel) {
                    @Override
                    public void handleDelivery(String consumerTag, Envelope envelope, AMQP.BasicProperties properties, byte[] body) throws IOException {
                        System.out.println("消费者1: " + new String(body));
                    }
                });
    }
}

# work quene模型

工作队列模型是一种用于在多个消费者之间分发和处理任务的方式。它的核心思想是将耗时的任务封装成消息发送到队列中，然后由多个消费者并发地从队列中取出任务并执行。

在上图的模型中，有以下概念：

P：生产者：任务的发布者

C1：消费者-1：领取任务并且完成任务，假设完成速度较慢

C2：消费者-2：领取任务并完成任务，假设完成速度快

# 案例场景

  • 假设有一个图像处理系统，用户上传大量的图片需要进行处理，处理过程耗时较长，可以使用 Work Queue 模式来平衡负载和提高处理效率。
  • 用户上传的图片会被放入一个任务队列中，多个消费者并行地从队列中获取任务并进行图片处理，以避免单一消费者负担过重。

# 消息自动确认机制

Doing a task can take a few seconds. You may wonder what happens if one of the consumers starts a long task and dies with it only partly done.With our current code,once RabbitMQ delivers a message to the consumer it immediately marks it for deletion. In this case, if you kill a worker we will lose the message it was just processing. We'llalso lose all the messages that were dispatched to this particular worker but were not yet handled.

But we don't want to lose any tasks. If a worker dies,we'd like the task to be delivered to another worker.

完成一项任务可能需要几秒钟的时间，您可能会想，如果消费者启动了一项长任务，但在完成之前就终止了，会发生什么。使用我们当前的代码，一旦RabbitMQ将消息传递给消费者，它就会立即将其标记为删除。在这种情况下，如果终止一个工作进程，它刚刚处理的消息就会丢失。发送给该特定工作者但尚未处理的消息也将丢失。

但我们不想失去任何任务。如果一名工人死亡，我们希望将任务交给另一名工人。

大白话解释：

当消息被全部发送给消费者时，如果消费者挂掉了，那么这个消息就会丢失，没有人来处理它了。为了避免这种情况，我们可以关闭自动确认机制，让消费者一条条地确认消息的处理。这样，消费者处理完一条消息后才会确认，然后再处理下一条消息。这样做的好处是，如果某个消费者挂掉了，那么未确认的消息会被重新发送给其他消费者，确保任务不会丢失。这种方式能够让工作能者多劳，保证所有的任务都能被及时处理。

package ack;

import com.rabbitmq.client.*;
import utils.RabbitMQUtils;

import java.io.IOException;
import java.util.concurrent.TimeoutException;

/**
 * @Description
 * @Author MrWei
 * @Date 2023/8/16 20:00
 **/
public class AckQueueConsumer2 {
    public static void main(String[] args) throws IOException{
        // 获取连接
        Connection connection = RabbitMQUtils.getConnection();
        final Channel channel = connection.createChannel();
        //每一次只能消费一个消息
        channel.basicQos(1);
        channel.queueDeclare("ack",true,false,false,null);

        // 参数1:队列名称
        // 参数2:消息自动确认[true:消费者自动向rabbitmq确认消息消费 false:不会自动确认消息]
        channel.basicConsume("ack",false,new DefaultConsumer(channel){
            @Override
            public void handleDelivery(String consumerTag, Envelope envelope, AMQP.BasicProperties properties, byte[] body) throws IOException {
                try{
                    Thread.sleep(2000);
                }catch (Exception e){
                    e.printStackTrace();
                }
                System.out.println("消费者-2: "+new String(body));
                // 手动确认
                // 参数1:确认队列中那个具体消息
                // 参数2:是否开启多个消息同时确实
                channel.basicAck(envelope.getDeliveryTag(),false);
            }
        });

    }
}

# Publish/Subscribe模型

RabbitMQ 的第三种模型是发布/订阅（Publish/Subscribe）模型。在这个模型中，一个消息发布者将消息发送给一个交换机（Exchange），交换机会将消息广播给所有与它绑定的队列，每个队列都有自己的消费者。这样，一个消息可以被多个消费者同时接收和处理

# 消息的发送流程

在`发布/订阅（Publish/Subscribe）模型下，消息的发送流程如下：

  1. 多个消费者：允许存在多个消费者，每个消费者都可以独立地监听消息。
  2. 队列分配：每个消费者拥有自己的队列，这样每个消费者都有一个独立的接收消息的通道。
  3. 绑定到交换机：每个队列都需要绑定到一个 Fanout 类型的交换机，交换机负责将消息广播给所有绑定的队列。
  4. 生产者发送消息：生产者将消息发送到交换机，而不是直接发送到队列。生产者无法决定消息被发送到哪个队列，这由交换机来决定。
  5. 交换机路由：交换机根据绑定关系，将消息广播给绑定在它上面的所有队列。
  6. 广播消费：每个队列中的消费者都能从自己的队列中获取相同的消息，实现了一条消息被多个消费者同时消费的效果。

# 案例场景

  • 假设有一个新闻发布系统，需要将新闻通知发送给不同的订阅者，同时也需要将新闻发布到网站和移动应用上。
  • 当有新的新闻发布时，系统将新闻消息发送到一个交换机（Exchange），绑定了不同队列的订阅者和发布系统会接收并处理这些消息，从而实现实时广播通知。

# Routing路由模型-Direct(直连)

在Fanout模式中，一条消息，会被所有订阅的队列都消费。 但是，在某些场景下，我们希望不同的消息被不同的队列消费。 这时就要用到Direct类型的Exchange。

在Direct模型下：队列与交换机的绑定，不能是任意绑定了，而是要指定一个RoutingKey（路由key）消息的发送方在 向Exchange发送消息时，也必须指定消息的 RoutingKey。

Exchange不再把消息交给每一个绑定的队列，而是根据消息的Routing Key进行判断，只有队列的Routingkey与消息的 Routing key完全一致，才会接收到消息.

在上图的模型中，有以下概念：

P：生产者，向Exchange发送消息，发送消息时，会指定一个routing key。 X：Exchange（交换机），接收生产者的消息，然后把消息递交给 与routing key完全匹配的队列 C1：消费者，其所在队列指定了需要routing key 为 error 的消息 C2：消费者，其所在队列指定了需要routing key 为 info、error、warning 的消息

# 案例场景

  • 假设有一个电商系统，不同的订单需要分别处理，且订单处理逻辑可能不同，可以使用 Routing 模式实现不同类型订单的有选择性路由。
  • 当有新订单生成时，根据订单的类型（普通订单、VIP订单等）将订单消息发送到交换机，不同队列的消费者根据绑定关系接收并处理特定类型的订单。

# Topic模型(动态路由模型)

Topic类型的Exchange与Direct相比，都是可以根据RoutingKey把消息路由到不同的队列。只不过Topic类型Exchange可以让队列在绑定Routing key的时候使用通配符！这种模型Routingkey 一般都是由一个或多个单词组成，多个单词之间以”.”分割，例如： item.insert

# 统配符语法

* (star) can substitute for exactly one word.    匹配不多不少恰好1个词
# (hash) can substitute for zero or more words.  匹配零个、一个或多个词

例如:

audit.#   匹配audit、audit.dbs 、或者audit.irs.abc
audit.*   只能匹配 audit.irs

# 案例场景

  • 假设有一个微服务架构的电影推荐系统，用户可以根据电影类型进行订阅，系统需要将符合用户订阅条件的电影推荐发送给用户。
  • 当有新电影上线时，系统将电影消息发送到交换机，并将消息的主题设置为电影类型（例如：科幻、爱情、动作等），订阅了对应类型的队列的用户会接收到相应的电影推荐。
